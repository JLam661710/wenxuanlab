# Vibe Coding 工程思维（一）：指挥官隐喻与交互心法

## 1. 核心观念重构：从“许愿”到“指挥”

很多初学者误以为 Vibe Coding 就是向“许愿池”投币（输入 Prompt），然后期待奇迹发生。这种心态导致了“一旦 AI 没做对，就束手无策”的局面。

**真正的 Vibe Coding 是指挥“超级工兵”作战。**

*   **你是导演（Director）**：你不需要知道摄影机光圈具体调到 F1.8 还是 F2.8（这是代码细节，AI 做）。
*   **AI 是工兵/摄像师**：它负责执行具体的技术实现。
*   **你的职责**：必须明确**这一场戏的情绪是什么（功能目标）**，**演员从哪里进场（数据来源）**，以及**如果不满意，该告诉摄像师“光太暗了”还是“构图歪了”（调试与反馈）**。

## 2. 结构化思维：搭积木 vs 捏泥巴

### 2.1 拒绝“捏泥巴”
错误的做法是试图用一段 500 字的长 Prompt 把整个应用一次性描述出来。这就像试图一次性捏出一个复杂的泥塑，一旦哪里塌了，只能全盘重来。

### 2.2 学会“搭积木”
工程思维要求我们将大功能拆解：
*   **拆解**：把应用拆成“登录页”、“商品列表”、“购物车”等独立组件。
*   **隔离**：修“购物车”的时候，不会弄坏“登录页”。
*   **指令范式**：“先帮我造一块红色的方积木（实现一个具体的小功能）”，而不是“给我变座城堡出来”。

## 3. 调试思维：报错是系统在求助

在 Vibe Coding 中，报错不是失败的标志，而是交互的开始。

*   **非工程思维**：看到报错 -> 恐慌 -> 截图发群 -> 等待救援。
*   **工程思维**：看到报错 -> 复制报错信息 -> 喂回给 AI -> 询问“这是什么意思？怎么修？”。

**具备工程本质的标志**：不再害怕报错，而是把报错当作与 AI 进行下一轮更精准对话的线索。调试本质上是你与 AI 协作修正“剧本”的过程。
