# 调试的艺术：从“Vibe”回归“Fact”

在 Vibe Coding 的创造阶段，我们依赖直觉、灵感和自然语言的流动（Vibe）；但在调试（Debugging）阶段，我们必须迅速切换回严谨的工程视角。

AI 不是心理咨询师，它不需要共情你的挫败感；AI 是逻辑精密的工程师，它需要的是“犯罪现场”的客观证据。

以下是向 AI Coding Agent 描述故障的四大黄金法则：

## 1. 法则一：保留“犯罪现场” (Raw Data Principle)
*   拒绝口语化加工：不要说“它挂了”、“它没反应”或者“它报错了”。
*   直接呈堂证供：直接复制 Terminal 或 Console 中的 红色报错信息（Stack Trace） 原文。
*   原理：AI 对错误代码（Error Code）和堆栈路径的索引能力远超人类。任何的人工转述都会导致关键信息的“熵增”和丢失。
    > Do: "这是控制台的报错信息：`TypeError: Cannot read properties of undefined (reading 'map') at Line 45...`"
    > Don't: "列表渲染不出来，好像是数据有问题。"

## 2. 法则二：构建“预期差” (The Gap Analysis)
一个完美的 Bug 描述应该包含三个要素，形成逻辑三角：
1.  Action（操作）：我具体做了什么？（点击了哪个按钮？输入了什么？）
2.  Expected（预期）：我本希望发生什么？
3.  Actual（现状）：实际发生了什么？
*   核心：通过对比“预期”与“现状”的差异，AI 能迅速锁定逻辑分叉点。

## 3. 法则三：主动增强“可观测性” (Observability First)
当错误不明显时，不要只问“为什么”，要懂得要求工具介入。
*   策略：主动要求 AI 在代码关键位置植入“探针”。
*   Prompt 范式：
    > "我不确定哪里出了问题，请帮我在数据请求前后加入 `console.log` 或详细的日志断点，我想看清楚数据流是在哪一步断掉的。"
*   价值：这不仅能解决当下的 Bug，还能让生成的代码具备更好的可维护性（Maintainability）。让 F12 开发者工具从一片空白变成信息丰富的仪表盘。

## 4. 法则四：坚守“上下文闭环” (Context Locality)
*   不要“脱离现场”：千万不要遇到报错就习惯性地复制去网页版的 ChatGPT/Claude 提问。
*   留在 IDE 里：Trae 等 AI IDE 的最大优势在于它拥有你的全量上下文（Context）——它知道你的文件结构、光标位置和依赖关系。
*   比喻：在 IDE 里提问是“现场勘查”，去外部网页提问是“电话问诊”。除非是纯理论问题，否则永远优先在开发语境（In-Context）中解决故障。

---

## 💡 核心技能：听懂 F12 的“暗语”
*   F12 (开发者工具) 不是黑客的专属，它是程序的“听诊器”。
*   学会打开 Console 面板，配合 法则三，你将不再是盲目地猜测，而是像外科医生一样精准地看到程序的“内脏”和“血流”。
