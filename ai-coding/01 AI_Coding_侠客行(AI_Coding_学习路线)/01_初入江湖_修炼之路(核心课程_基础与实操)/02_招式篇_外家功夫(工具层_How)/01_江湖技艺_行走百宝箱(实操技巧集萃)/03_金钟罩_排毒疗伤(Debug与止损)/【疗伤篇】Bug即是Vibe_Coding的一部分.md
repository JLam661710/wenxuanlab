# 教学设计：Bug 即是 Vibe Coding 的一部分

## 1. 核心理念：红字不是“失败”，是“对话”

在传统的应试教育里，红叉（❌）代表错误，代表扣分，代表“你不行”。
但在 Vibe Coding 中，红色的报错信息（Error Message）是计算机在拼命地试图告诉你线索。

*   心态重塑：看到报错不要慌，不要觉得“我搞砸了”。要觉得是“系统在跟我说话，只是它说话比较大声（红色），而且有点结巴（只有代码）。”
*   教学目标：消除对红字的生理性恐惧（Fight or Flight），建立“这是解谜线索”的游戏化心态。

## 2. 教学环节设计：故意报错（Intentional Bugging）

我们可以设计一个“找茬游戏”，带学员亲手制造并修复三个经典的“劝退级”错误。

### 场景一：语法的“洁癖” —— JSON 逗号丢失案
*   操作：让学员在 JSON 的某一行末尾故意删掉一个逗号 `,`。
*   现象：程序直接炸了，可能报 `SyntaxError: Expecting ',' delimiter`。
*   解读：
    *   告诉学员：计算机是有洁癖的。在它的世界里，少一个逗号就像句子没说完。
    *   Vibe Check：看报错行号（Line Number）。它通常会精确地指向案发现场（或者下一行）。

### 场景二：拼写的“马虎” —— 消失的变量案
*   操作：把 `print(user_name)` 故意写成 `print(username)`（少了个下划线）。
*   现象：`NameError: name 'username' is not defined`。
*   解读：
    *   告诉学员：计算机记性不好，它只认识你之前正式介绍（定义）过的词。你没介绍过，它就不认识。
    *   Vibe Check：看到 `not defined`，立刻去查拼写。不要怀疑是电脑坏了，99.9% 是我们手滑了。

### 场景三：类型的“固执” —— 数字与文字的联姻失败案
*   操作：尝试把 `"100"` (字符串) 和 `1` (数字) 相加。
*   现象：`TypeError: can only concatenate str (not "int") to str`。
*   解读：
    *   告诉学员：在 Python 里，文字和数字是两个物种，不能直接结婚。你得先给它们“整容”（类型转换，`int()`）。
    *   Vibe Check：看到 `TypeError`，就是物种隔离问题。

## 3. 升华：Debugging 是最高级的 Vibe

*   高光时刻：当学员根据报错信息，自己把那个逗号加上，或者把变量名改对，然后按下运行，红字消失，绿色的结果跳出来的那一刻。
*   金句：“编程不是一次性写出完美的代码，编程是不断地犯错、修正、再犯错、再修正的过程。每一个被你修好的 Bug，都是你的一枚勋章。”

通过这种“先破坏，再修复”的脱敏疗法，学员再遇到自然发生的 Bug 时，第一反应就不会是“我不行”，而是“哦，我又触发了那个逗号陷阱，看我怎么抓到它”。
