### 5. 被植入的先验：我们是如何被“格式化”的？

在开始敲下第一个 Prompt 之前，我们的思维就已经被预设、被固化、甚至被隐形植入了一套**“技术先验”（Technological A Priori）**。如果不反思这些先验，我们就无法真正理解我们在做什么。

#### (1) GUI 先验（Graphical User Interface A Priori）
*   **位置**：第一层（机器/OS）
*   **预设**：**“计算必须是可视化的、桌面化的。”**
*   **实质**：操作系统让我们产生了一种错觉，以为文件就是一个“文件夹图标”，程序就是一个“窗口”。这种先验剥夺了我们对计算机**线性流（Stream）**和**文本性（Textuality）**的感知。
*   **后果**：Vibe Coder 往往难以理解 CLI（命令行），因为 CLI 破坏了 GUI 先验。他们习惯了点点点，而不是输入指令。

#### (2) 对象化先验（Objectification A Priori）
*   **位置**：第二层（代码/语言）
*   **预设**：**“世界必须被抽象为对象（Object）和类（Class）。”**
*   **实质**：现代编程语言（特别是面向对象编程 OOP）强迫我们将流动的现实世界切割成一个个独立的、模块化的“积木”。
*   **后果**：Vibe Coder 在与 AI 交流时，往往会被 AI 输出的代码结构所规训。AI 会倾向于生成符合 OOP 规范的代码，从而潜移默化地塑造了用户的思维方式——凡事都要封装，都要模块化，即使有时候一个简单的脚本更有效。

#### (3) 应用化先验（App-ification A Priori）
*   **位置**：第三层（应用/LLM）
*   **预设**：**“解决问题的终极形式必须是一个 App。”**
*   **实质**：这是移动互联网时代最大的意识形态植入。它让我们遗忘了，计算机本质上是一个**通用计算机器（Universal Turing Machine）**，而不是一个“App 播放器”。
*   **后果**：Vibe Coding 极其容易陷入“造轮子”的陷阱。用户会为了一个极小的需求（比如统计一下字数），去让 AI 写一个带 GUI 界面的 Python App，而不是直接运行一行代码。**“应用化先验”让我们丧失了直接计算的能力。**
